# 자동화 도구 비교 구현 (Make vs Zapier)[cite: 2]

**워크플로우:** "Google Sheets 새 행 추가 → 조건 분기(점수 기준) → Discord 알림 발송 + Gmail 전송"[cite: 2]

### [Make 구현][cite: 2]
- **Trigger:** Google Sheets – Watch New Rows[cite: 2]
- **Router:** 점수 80점 이상 / 미만 분기[cite: 2]
- **Action 1:** Discord – Send a Message (합격 알림)[cite: 2]
- **Action 2:** Gmail – Send an Email (합격 축하 메일)[cite: 2]

### [Zapier 구현][cite: 2]
- **Trigger:** Google Sheets – New Spreadsheet Row[cite: 2]
- **Filter:** 점수 80점 이상만 통과 (COL B Greater than 79)[cite: 2]
- **Action 1:** Discord – Send Channel Message[cite: 2]
- **Action 2:** Gmail – Send Email[cite: 2]

---

### 구현 과정 요약[cite: 2]
- **데이터 소스 정의:** Google Sheets를 트리거로 설정하여 새 행이 추가될 때마다 학생 성적 데이터(이름, 점수, 이메일)를 자동으로 감지하도록 구성함.[cite: 2]
- **조건 분기 설정:** '점수' 필드를 기준으로 80점 이상인 경우에만 후속 작업이 실행되도록 필터링함.[cite: 2]
- **다중 액션 실행:** 조건 만족 시 하나의 트리거로부터 Discord 채널 메시지 발송과 Gmail 축하 이메일 전송이라는 2개 이상의 서로 다른 액션이 순차적으로 실행되도록 워크플로우를 완결함.[cite: 2]
- **도구별 제약 우회:** Make에서는 내장 기능인 Router를 사용하여 다중 분기를 직관적으로 처리하였으며, Zapier 무료 플랜에서는 다중 분기(Paths) 제약을 우회하기 위해 Filter 모듈을 도입하여 조건 불충족 데이터를 차단하는 방식으로 동일한 결과물을 도출함.[cite: 2]

| 1 | UI/UX | 시각적인 동그라미(노드) 마인드맵 형태 구조. 드래그 앤 드롭으로 복잡한 다차원 흐름을 자유롭게 조망하기 유리함.[cite: 2] | 위에서 아래로 단방향으로 내려오는 리스트 형태 구조. 인터페이스가 정돈되어 있어 텍스트 중심 가독성이 높음.[cite: 2] |
| 2 | 설정 난이도 | 자유도가 높은 대신 각 모듈 간의 세부 데이터 매핑 및 데이터 타입 지정 시 초기 학습 곡선이 다소 존재함.[cite: 2] | 단계별 설정 가이드 마법사가 매우 친절하여 초보자도 마우스 클릭 몇 번으로 쉽게 연동을 마칠 수 있음.[cite: 2] |
| 3 | 연동 서비스 범위 | 글로벌 주요 서비스 연동을 넓게 지원하나, 마이너한 앱의 경우 웹훅(Webhook)이나 직접 API 설정이 필요할 수 있음.[cite: 2] | 현존하는 자동화 도구 중 가장 방대한 양의 서드파티 앱과 공식 커넥터를 보유하고 있어 앱 생태계 접근성이 매우 뛰어남.[cite: 2] |
| 4 | 무료 플랜 범위 | **매월 1,000번의 실행 수(Ops)**를 기본 제공하여 복잡하고 잦은 연동 테스트 및 실무 적용 시 비용 부담이 적음.[cite: 2] | **매월 100번의 실행(Tasks)**만 제공하며 다단계 액션 설계 시 횟수가 빠르게 차감되어 무료 플랜의 제약이 큼.[cite: 2] |
| 5 | 실행 로그 확인 방식 | 실행 히스토리 맵에서 각 노드를 마우스로 클릭해 단계별 입력(Input) 및 출력(Output) 데이터를 시각적으로 정밀 디버깅 가능함.[cite: 2] | Task History 메뉴에서 텍스트 기반의 단계별 성공/실패 로그 리스트를 제공하여 깔끔하게 에러 발생 지점을 파악할 수 있음.[cite: 2] |

### 각 도구의 장단점 정리[cite: 2]

#### 1) Make[cite: 2]
- **장점:** 무료 플랜의 혜택(1,000 Ops)이 크고, 다중 조건 분기(Router)를 개수 제한 없이 무료로 설계할 수 있어 고차원 자동화에 유리함.[cite: 2]
- **단점:** 초기 인터페이스 구조가 다소 복잡하여 툴에 익숙해지는 데 시간이 걸림.[cite: 2]

#### 2) Zapier[cite: 2]
- **장점:** 인터페이스가 극도로 직관적이며, 전 세계 거의 모든 서드파티 앱과의 커넥터를 공식 지원함.[cite: 2]
- **단점:** 무료 플랜의 실행 횟수가 너무 적고, 다중 분기(Paths) 기능이 유료 등급에만 묶여 있어 우회 설계가 필요함.[cite: 2]

---

### 어떤 상황에서 적합한지 의견[cite: 2]

- **Make가 적합한 상황:** 비용을 들이지 않고 무료 플랜 내에서 복잡한 다단계 조건 분기(Router)나 대규모 데이터 매핑이 필요한 정교한 자동화 아키텍처를 구축하고자 할 때 가장 적합함.[cite: 2]
- **Zapier가 적합한 상황:** 복잡한 기능 제약이나 학습 시간 없이, 대중적인 앱들을 빠르게 결합하여 단방향성(일직선 구조)의 직관적인 프로토타입 자동화를 초고속으로 구축하고 검증하고 싶을 때 적합함.[cite: 2]

## [프로젝트 2] 자유 주제 자동화 설계 및 구현 (Make)[cite: 2]

### 1) 자동화할 반복 업무 정의[cite: 2]
- **반복 업무:** 매일 포털 사이트나 날씨 앱에 수동으로 접속하여 현지(영국 런던)의 기상 정보를 검색하고 확인하는 일상적인 루틴.[cite: 2]
- **자동화 목적:** 매일 지정된 시각에 영국 런던 지역의 날씨 데이터를 자동으로 수집하여 주로 사용하는 협업 도구인 Discord 채널로 전송받음으로써, 정보 탐색에 드는 불필요한 행동과 시간을 최소화함.[cite: 2]

### 2) 도구 선정 및 선정 이유[cite: 2]
- **선정 도구:** Make (메이크)[cite: 2]
- **선정 이유:** 별도의 외부 날씨 API 키 발급 신청 및 복잡한 인증 절차 없이도 기본 제공되는 'Weather' 내장 모듈을 통해 전 세계 날씨 데이터를 즉시 호출할 수 있음. 또한, 매일 정해진 시각에 작동하도록 예약하는 스케줄러 기능과 넉넉한 무료 플랜 범위(월 1,000회 실행)를 제공하여 무인 자동 알림 시스템을 상시 가동하기에 가장 최적의 선택지임.[cite: 2]

### 3) 자동 실행 구조 구현[cite: 2]
- **자동 실행 조건:** 매일 오후 10시에 자동으로 전체 시나리오가 트리거되어 작동하도록 내부 스케줄러 세팅을 완료함.[cite: 2]

### 4) 워크플로우 흐름 설명[cite: 2]
- **Trigger (스케줄 기반 날씨 데이터 호출):** 설정해 둔 매일 오후 10시 정각이 되면 Weather - Get Current Weather 모듈이 자동으로 실행됨. 이 단계에서 미리 설정된 영국 런던 지역(london, uk)의 현재 기온 및 기상 데이터를 실시간으로 API 서버에서 호출함.[cite: 2]
- **Action (디스코드 메시지 전송):** 트리거 단계에서 긁어온 데이터 중 현재 기온과 날씨 상태값을 추출함. 이를 디스코드 메시지 전송 모듈로 전송함.[cite: 2]
